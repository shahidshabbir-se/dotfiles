# Investigate dim Chromium on Hyprland

## What the current config suggests
- `modules/hyprland.nix:27` enables desktop-only HDR/color management on the monitor with `bitdepth,10,cm,hdr,sdrbrightness,1.25,vrr,0`.
- `modules/hyprland.nix:205` also enables compositor color management globally with `render.cm_enabled = true`.
- This combination is the strongest explanation for "Chromium looks dim while other apps look normal".

## Why this happens
- Recent Chromium builds on Wayland enable `WaylandWpColorManagerV1` by default.
- Under Hyprland HDR / color-managed outputs, Chromium can opt into the Wayland color-management path and render as HDR-aware content.
- When that happens, Hyprland SDR boost settings such as `sdrbrightness = 1.25` may no longer affect Chromium the same way they affect normal SDR apps, making Chromium appear dim/darker.
- This matches current Hyprland/Chromium reports for `cm,hdr`, `cm,edid`, and `cm,wide` outputs.

## Planned verification
1. Confirm whether the dim app is actually `chromium` and not `zen-beta`.
2. Check whether Chromium is running native Wayland mode on this machine.
3. Compare behavior with color-management disabled only for Chromium first, before changing Hyprland globally.

## Planned fixes in preferred order
1. Chromium-only workaround
- Add Chromium launch flags to disable the Wayland color-management protocol:
- `--disable-features=WaylandWpColorManagerV1`
- Preferred place: `~/.config/chromium-flags.conf` or a Home Manager-managed equivalent if one already exists in this repo.
- This is the smallest and safest fix because it keeps the rest of the HDR/Hyprland setup unchanged.

2. If needed, make Chromium use X11 instead of native Wayland
- Fallback launch flag:
- `--ozone-platform=x11`
- This is less desirable than step 1 because it gives up native Wayland behavior.

3. If the user wants a compositor-side fix instead of app flags
- Reduce or disable HDR/color management on the monitor path:
- change `cm,hdr` to `cm,srgb` or remove HDR-specific monitor params for normal desktop use
- or disable `render.cm_enabled`
- This is broader impact and should only be used if the user wants to trade away HDR/color-managed desktop output.

## Planned implementation path
1. Search the repo for existing Chromium/Chrome flag management.
2. If none exists, add the minimal Home Manager config for Chromium flags.
3. Rebuild/apply Home Manager config.
4. Re-test Chromium brightness.
5. Only if still dim, apply fallback step 2.

## Expected outcome
- Chromium should match the perceived brightness of other non-color-managed apps again.
- Hyprland HDR settings can remain enabled for the rest of the desktop unless the user explicitly wants a compositor-wide change.
